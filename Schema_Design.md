# Schema Design — FinTech Data Pipeline

## Project Overview

End-to-end financial data pipeline built on the PaySim synthetic
fraud dataset (6.3M transactions). Designed to demonstrate
enterprise-grade data architecture for fraud detection.

## Dataset

* Source: PaySim — Kaggle (kaggle.com/datasets/ealaxi/paysim1)
* Rows: 6,362,604 transactions
* Period: 30 days (744 hourly steps)
* Fraud rate: \~0.13% (highly imbalanced — intentional)

## Database

* Engine: SQLite (development) → PostgreSQL (production, Phase 2)
* Tool: DBeaver

\---

## Tables

### 1\. raw\_transactions

Primary data table. One row per financial transaction.

|Column|Type|Notes|
|-|-|-|
|transaction\_id|INTEGER|Primary key, auto-generated|
|step|INTEGER|1 step = 1 hour (1–744)|
|type|TEXT|Transaction category|
|amount|REAL|Transaction value|
|nameOrig|TEXT|Sender account ID|
|oldbalanceOrg|REAL|Sender balance before|
|newbalanceOrig|REAL|Sender balance after|
|nameDest|TEXT|Receiver account ID|
|oldbalanceDest|REAL|Receiver balance before|
|newbalanceDest|REAL|Receiver balance after|
|isFraud|INTEGER|1 = fraud, 0 = legitimate|
|isFlaggedFraud|INTEGER|1 = system flagged, 0 = not|
|loaded\_at|DATETIME|Auto-set on insert|

### 2\. ingestion\_logs

Audit trail for every data load run.

|Column|Type|Notes|
|-|-|-|
|log\_id|INTEGER|Primary key, auto-generated|
|run\_date|DATETIME|Auto-set on insert|
|rows\_loaded|INTEGER|Successfully inserted rows|
|rows\_rejected|INTEGER|Rows dropped during cleaning|
|status|TEXT|SUCCESS / FAILED / PARTIAL|
|duration\_secs|REAL|Total load time in seconds|
|notes|TEXT|Free text — chunk count, errors|

### 3\. transaction\_categories

Reference/lookup table for transaction types and risk levels.

|Column|Type|Notes|
|-|-|-|
|category\_id|INTEGER|Primary key, auto-generated|
|type\_code|TEXT|Unique transaction type|
|description|TEXT|Human-readable explanation|
|risk\_level|TEXT|LOW / MEDIUM / HIGH|

## Transaction Type Analysis (Full Dataset)

|Type|Count|% of Total|Risk Level|
|-|-|-|-|
|CASH\_OUT|2,237,484|35.17%|HIGH|
|PAYMENT|2,151,495|33.81%|LOW|
|CASH\_IN|1,399,284|21.99%|LOW|
|TRANSFER|532,909|8.38%|HIGH|
|DEBIT|41,432|0.65%|MEDIUM|

Key insight: CASH\_OUT + TRANSFER = 43.55% of all transactions
and are the two highest fraud-risk types. This drives the
fraud detection logic in Phase 5.

## Design Decisions

1. **Surrogate key added**: transaction\_id does not exist in the
source CSV. Auto-generated by SQLite on insert. This is
intentional — raw datasets rarely come with reliable unique IDs.
2. **camelCase column names kept**: Raw table uses original CSV
naming (nameOrig, isFraud etc.) to simplify ingestion.
Renamed to snake\_case during PostgreSQL migration in Phase 2.
3. **name columns are nullable**: NOT NULL enforcement on account
name columns happens at the ETL layer (Phase 2 Python script),
not at the raw table level. Standard data warehouse practice.
4. **Chunked ingestion**: Full dataset loaded in 50,000-row chunks
to prevent memory overflow. Load time: \~2–5 minutes for 6.3M rows.
